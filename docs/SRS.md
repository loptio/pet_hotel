# 軟體需求規格書（SRS）— 寵物旅館與美容預約系統

> 版本 v1.0　|　基準來源：軟體設計作業之 User Requirements（FR/NFR 為權威原文）
> 本文件是後續系統實作的需求基準；設計請見 `SDD.md`。

---

## 1. 簡介

### 1.1 目的
提供寵物旅館（住宿）與美容預約的一站式線上服務：協助飼主完成線上預約、查詢、付款與進度追蹤；協助店家完成入住管理、美容流程管理與通知回報，降低人工溝通成本、提升服務品質。

### 1.2 範圍
**涵蓋**：帳號與權限、寵物與健康檔案、預約與資源（床位／時段）管理、服務執行（報到、美容工作單）、付款與退款、通知、稽核、例外處理（緊急事件、疫苗逾期、未到場）。
**不涵蓋**：金流與推播之內部實作（由第三方 ECPay、FCM／APNs 提供）、實體 POS 機。

### 1.3 角色（Actors）
| 角色 | 職責 |
|---|---|
| 飼主 Owner | 建立寵物檔案、建立／取消預約、付款、查詢進度與照片 |
| 櫃台人員 Front Desk | 報到核驗、床位管理、審核中度危險預約、標記危險等級（低／中） |
| 美容師 Groomer | 執行美容工作單、更新四階段狀態、上傳照片、觸發緊急事件 |
| 系統管理員 Admin | RBAC 管理、帳號封鎖、標記／解除高度危險、查異常取消報告 |
| 金流服務 ECPay〔外部〕 | 第三方支付：授權、扣款、退款 |
| 推播服務 FCM／APNs〔外部〕 | 第三方推播通知 |

### 1.4 名詞定義
- **危險等級 DangerLevel**：None／Low／Medium／High，決定預約管控強度。
- **床位 Kennel**：實體住宿單位，具唯一編號、類型（標準／豪華）、狀態（空床／已預約／已入住／清潔中）。
- **工作單 WorkOrder**：美容服務的執行單，含預檢→洗澡→烘乾→剪毛四階段。
- **訂金 Deposit**：預估總金額之 30%。
- **稽核日誌 Audit Log**：不可竄改之操作紀錄。

## 2. 功能需求（Functional Requirements）

> 條文為作業之權威原文，僅依模組重新分組（原文中 FR-03.1 誤接於 FR-02.8 段尾，此處已拆回 FR-03）。

### FR-01　帳號與權限
- **FR-01.1** 系統應提供使用者註冊、登錄及基於電子郵件之密碼重設功能。
- **FR-01.2** 系統應根據預設角色（飼主、櫃台、美容師、管理員）實施嚴格之角色存取控制（RBAC）。
- **FR-01.3** 系統應提供管理員查詢異常取消報告之界面，並允許管理員手動封鎖指定帳號。
- **FR-01.4** 系統應允許所有已登入使用者檢視及編輯個人基本資料，包含姓名及聯絡電話。

### FR-02　寵物與健康檔案
- **FR-02.1** 系統應允許飼主建立並編輯寵物基本檔案，包含名稱、品種、年齡。
- **FR-02.2** 系統應允許飼主新增寵物醫療背景紀錄，但不得修改或刪除既有紀錄。
- **FR-02.3** 系統應允許飼主在寵物檔案中預先錄入疫苗種類及有效截止日期，此為選填資料。
- **FR-02.4** 系統應要求飼主在預約完成後上傳動物醫院開立之疫苗證明文件。
- **FR-02.5** 系統應允許櫃台人員於報到時核對晶片號碼與文件內容，確認無誤後將疫苗記錄錄入系統，系統於錄入後自動校驗有效期（見 FR-06.1）。
- **FR-02.6** 系統應允許飼主在寵物檔案中填寫行為備註（如怕生、不喜歡被碰腳等），此為選填資料。
- **FR-02.7** 系統應允許櫃檯人員及美容師對寵物標記危險等級（低度／中度）並附說明；管理員可標示高度危險等級及解除封鎖。
- **FR-02.8** 系統應依寵物危險等級執行對應的預約管控：無標記或低度僅於報到介面顯示警告；中度預約建立後狀態標記為待審核，須由櫃台人員審核通過方可確認；高度系統應自動拒絕預約並顯示提示，須由管理員解除封鎖後方可再次預約。

### FR-03　預約與資源
- **FR-03.1** 系統應允許飼主選擇特定寵物、服務類型及預約時段。
- **FR-03.2** 系統應在預約提交時即時驗證房位、人力及時段之可用性，若已被他人佔用則拒絕並提示飼主重新選擇，**嚴禁超額預約**。
- **FR-03.3** 系統應允許飼主於任何時間取消預約。
- **FR-03.4** 系統應允許飼主查詢目前及歷史預約紀錄，包含預約狀態與服務類型。
- **FR-03.5** 系統應於預約時段開始後 2 小時內若未完成報到，自動將預約狀態標記為「未到場」並釋放對應資源。
- **FR-03.6** 系統應支援實體床位管理：每個床位有唯一編號、類型（標準／豪華）及狀態（空床、已預約、已入住、清潔中）；系統應於預約建立時依房型自動分配合適床位，並即時顯示哪個床位住著哪隻寵物。
- **FR-03.7** 系統應支援服務項目分級：住宿分標準房與豪華房，美容分基礎美容與完整美容，各級定價不同。

### FR-04　服務執行與通知
- **FR-04.1** 系統應提供櫃台人員核驗預約狀態及疫苗有效性之報到界面。
- **FR-04.2** 系統應允許美容師切換服務狀態（待處理、進行中、已完成）並上傳作業照片。
- **FR-04.3** 系統應於服務狀態變更時，透過推送自動通知關聯飼主。
- **FR-04.4** 系統應提供飼主查詢目前預約服務執行狀態及美容師上傳之作業照片。
- **FR-04.5** 美容服務工作單應包含預檢、洗澡、烘乾、剪毛四個階段，美容師可逐步更新每階段狀態，系統應於每階段完成時推送通知飼主。

### FR-05　付款與稽核
- **FR-05.1** 系統應記錄每筆交易之應收金額、實付金額、支付管道及時間戳記。
- **FR-05.2** 系統應產生不可竄改之操作紀錄（Audit Log），涵蓋：預約建立與取消、報到成功與阻斷、服務狀態變更、付款交易、緊急事件觸發，並記錄操作人與時間戳記。
- **FR-05.3** 系統應於預約建立時向業主收取預估總金額 **30% 作為訂金**；餘款應於報到當日營業結束前結清，飼主可選擇線上付款或現場刷卡。
- **FR-05.4** 飼主於預約開始前 **24 小時以上**取消者全額退還訂金；**24 小時內**取消或未到場者訂金不予退還。

### FR-06　例外與自動化
- **FR-06.1** 系統應於辦理入住時自動校驗疫苗日期，若已逾期則必須暫停報到、待補件完成並顯示異常提示。
- **FR-06.2** 系統應於飼主選擇時段時，僅顯示當前可用之空閒時段，不顯示已額滿之時段。
- **FR-06.3** 系統應允許工作人員（櫃台及美容師）手動觸發「緊急醫療事件」流程，並強制要求錄入事件描述。
- **FR-06.4** 系統應於床位入住完成後自動更新床位狀態為已入住，寵物離開後自動更新為清潔中，清潔完成後由櫃檯人員手動標記為空床。

## 3. 非功能需求（Non-Functional Requirements）
- **NFR-01（可用性）** 90% 以上之核心業務流程（如預約、報到）能在 3 次點擊內觸達目標頁。
- **NFR-02（效能）** 並行 50 人以下時，所有查詢介面回應時間 < 3 秒。
- **NFR-03（安全性）** 對手機號碼、身份證號等敏感資訊去識別化；**付款資訊不得儲存於系統內部**，須全程由第三方金流處理。
- **NFR-04（可靠性）** 採交易機制確保支付狀態與服務狀態同步更新，一致性成功率 100%。
- **NFR-05（防濫用）** 每筆取消操作完整記錄時間與原因，保存期限 ≥ 180 天。
- **NFR-06（可維護性）** 架構採模組化設計，新增服務類型或金流供應商時不需修改核心預約邏輯。

## 5. 關鍵業務規則（衍生規則，實作必守）
1. **訂金＝預估總額 30%**；尾款於報到當日結清，付款方式僅「線上」或「現場刷卡」（**不收現金為訂金**）。
2. **退款 24h 規則**：開始前 ≥24h 取消全額退；<24h 或未到場不退（FR-05.4）。
3. **未到場**：時段開始後 2h 未報到 → 狀態 NoShow，自動釋放床位／時段資源（FR-03.5）。
4. **危險等級管控**：高度=自動拒絕（須 Admin 解封）；中度=待審核（櫃台核可）；低／無=報到時警告（FR-02.8）。
5. **疫苗有效性**：報到時自動校驗，逾期則暫停報到待補件（FR-06.1）。
6. **床位狀態循環**：空床→已預約→已入住→清潔中→（櫃台手動）空床（FR-03.6／06.4）。
7. **防超額預約**：提交時即時驗證並鎖定資源，嚴禁同資源重複佔用（FR-03.2）。
8. **稽核不可竄改**：Audit Log 僅追加、保存 ≥180 天（FR-05.2／NFR-05）。
